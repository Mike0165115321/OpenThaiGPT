## 1. Trade-offs Analysis (การวิเคราะห์ทางเลือก แลกได้แลกเสีย)

ความจริงที่เจ็บปวดกระดองใจที่สุดในโลก AI คือ **เราได้ทุกอย่างไม่ได้** ทุกการตัดสินใจมีทรัพยากรที่ต้องจ่าย (Memory, Disk, Latency)

### 1.1 RAG Embedder: CPU vs GPU
คำถามที่พบบ่อย: *ในเมื่อ GPU มันเร็ว ทำไมถึงบังคับให้ `BAAI-bge-m3` ทำงานดึง RAG บน CPU ล่ะ?*

**คำตอบ:** (Trade-off: Latency vs Model Size)
*   การ์ดจอเรามีขนาดจำกัดจำเขี่ย (สมมติว่า 8GB VRAM)
*   โมเดลภาษา 7B แบบ 4-bit กินไปแล้ว ~6GB
*   เมื่อเกิดการ Query, ค่า Context Memory กินเพิ่มอีก ~1.5GB
ยอดรวมก็ปริ่มทะลุเพดาน OOM (Out Of Memory) ขอบฟ้าแล้ว ถ้าเราเอาเดอะแบกอย่าง `bge-m3` (กิน VRAM 1-2GB) ลงไปยัดอัดด้วย ก็เท่ากับระบบล่ม 100% 

**สถาปัตยกรรมทางออก:** แยกความรับผิดชอบ (Asymmetric Hardware Workload): 
- ให้ LLM ซึ่งเป็นจุดคอขวดที่สุดเรื่องความเร็ว ทำงานบน GPU วีไอพี (VIP Access)
- ให้ Embedder ทำงานประมวลบน CPU ถึงแม้การเปลี่ยน Text → Vector บน CPU จะใช้เวลาช้ากว่าเสี้ยววินาที (ประมาณ 0.3s เทียบกับ 0.05s บน GPU) แต่เรา **ซื้อการรอดชีวิตของโปรแกรม (Stability)** แลกความอืดไปเพียง 0.25 วินาที — การแลกเปลี่ยนที่คุ้มโครตๆ!

### 1.2 การตัดโมฆะ Cross-Encoder Reranker
ในแพลนวิจัยมักจะบอกว่า RAG ที่ดีคือ `Retriever + Reranker` ทำไมงานนี้ผมถึงตัด Reranker ทิ้ง?

**คำตอบ:** 
*   **การศึกษา 1:** โมเดล BGE-M3 มีความตระหนักรู้ต่อความต่างของภาษาไทยที่สูงเกือบๆ จะทาบรัศมี Rerank ได้เลย 
*   **คอขวดของ Reranker:** Reranking ปกติบังคับเอาคำตอบมาจับคู่ Cross-attention ซึ่งกิน Overhead สูง ทอนเวลารอคอย API บวกเพิ่มไปอีก 2-4 วินาที 
*   **บทสรุป:** เราเน้น UX ที่ "ผู้รับสารไม่ต้องรอจนหลับ" ในบริบทยูสเคสหนังสือฐานข้อมูลเล็ก จึงตัดออกเพื่อประหยัดทรัพยากรคอมพิวเตอร์และเร่งความเร็ว

## 2. Asynchronous Threading Strategy (ศัลยกรรมลดคอขวดคิว API)

ใน `main.py` สังเกตว่าเราใช้ `run_in_threadpool(ai_core.generate_response, question.text)` ตลอดเลย ทั้งๆ ที่ FastAPI รองรับ `async def` แท้ๆ

**ไขรหัสลับเบื้องหลังฮาร์ดคอร์:** 
โมเดล HuggingFace `.generate()` เป็น **Synchronous Blocking I/O Process**! กล่าวมันคือการยึดบัลลังก์ Thread ตัวหลัก (Event Loop) ทรยศต่อกฎหมายของ FastAPI หากปล่อยมันรันตรงๆ เว็บไซต์ตัวนี้ถ้ามีคนทักพร้อมกัน 2 คน คนที่ 2 จะเจออาการ "ปุ่มค้าง โหลดไม่หมุน" แข็งทื่อจนคนแรกจะเจนิตข้อความจบ

เราใช้คำสั่ง `run_in_threadpool` ย้ายงานสร้าง Text สยองขวัญหนักๆ โยนไปปัดฝุ่นลง Thread worker (Thread ของระบบที่เตรียมไว้) ตัว Event Loop ของ Server ให้อยู่ว่างๆ ไว้รอรับ HTTP Connection ให้พร้อม ส่งผลให้ Server สามารถเสก "สถานะการทำงาน" เช่น การส่าย Request หรือตอบ Chat API ได้ 1,000 requests ต่อวินาทีถึงแม้ Backend จะค้างก็ตาม ทำให้ระบบสามารถ Scale รอพร้อมระบบคิวไหวในอนาคต

## 3. Algorithm: Strict Curatorial Prompt Injection

เมื่อโหมดคิวเรเตอร์ (Curator Mode) เปิดตัว คุณจะพบการใช้เทคนิค **Strict Formatting Constraints**:
```text
[กฎสำคัญที่สุด]
- ห้ามใช้หัวข้อกำกับใดๆ ทั้งสิ้นในคำตอบของคุณ
```
**การวิเคราะห์ทางจิตวิทยา LLM:** 
LLM พยายามเป็น "เด็กดี" เสมอ โดยเมื่อเจอ Context จำนวนเยอะ ตัวมันจะคิดว่าต้องจัดย่อหน้าให้ยาวที่สุดใส่ Bullet Point แจกข้อๆ เราต้องบังคับด้วย Prompt Layer ให้มันหลอมละลาย (Melt) แนวคิดออกมาเป็น Essay ให้ผู้รับสารรู้สึกว่า "กำลังคุยกะนักปราชญ์ที่รู้รอบด้าน ไม่ใช่พนักงานค้นเอกสาร"

## 4. The Scale Out Plan (วิเคราะห์สเกลหากมีผู้ใช้แตะ 10,000 คน)

การออกแบบทั้งหมดที่ทำมาบน Local เครื่องเดียว หากจะดันไปสู่ระดับ Global Scale, จุดคอขวดที่เราจะสังหารไปทีละตอคือ:

1. **คอขวด FAISS in-memory:** 
   *(ปัจจุบัน)* โหลด FAISS ขึ้น RAM เครื่อง (โอเคสำหรับข้อมูลหลักพัน Record)
   *(อนาคต)* อพยพไป VectorDB แยกเซิร์ฟเวอร์ เช่น Qdrant หรือ Milvus
2. **คอขวด Generator GPU Sharing:** 
   *(ปัจจุบัน)* GPU ควบรวมคำนวณ Sequential Request
   *(อนาคต)* ติดตั้ง TGI (Text Generation Inference) ของ Hugging Face ทำ Batching requests สร้างข้อความได้ทีละ 100 คนพร้อมกัน 
3. **คอขวด Session:**
   *(ปัจจุบัน)* `ai_core.chat_history` จำได้แค่สติคนๆ เดียว (Singleton pattern)
   *(อนาคต)* โยก History เข้า Redis Database ล็อก User ID ต่อ 1 คีย์
